# Design: 写作工具新增「读者视角」+「找引文」

**日期**: 2026-06-18
**状态**: 待实现
**范围**: 写作页写作工具栏目下，与现有四个工具（缩减/同义/比喻/扩展）并列，新增两个子功能：「读者视角」（请一位人格读者读完整篇、回一封信）与「找引文」（为选中论断联网检索可信证据）。

---

## 1. 背景与目标

### 现状

写作页右侧「✍️ 写作工具」面板（`frontend/src/components/AssistPanel.jsx`）当前提供四个**无状态文本变换**：缩减、同义替换、比喻、扩展。它们共享同一形状——**选中一段 → AI 返回文字 → 替换/复制原文**：

- 前端只传 `{ text, context }`，后端 `_assist_call`（`backend/main.py`）统一处理，默认从库里读 SOUL 风格文档注入 system（比喻例外，显式绕过 SOUL）。
- 两类输出：single（缩减/扩展，返回一段文字，可「采用替换」）、options（同义/比喻，返回多候选，点选采用或复制）。
- 模型分层：缩减用 Haiku，同义/扩展用 Sonnet，比喻用 Opus。

缺口：现有工具都在"帮你改句子"，没有"帮你看整篇"的反馈型工具，也没有"为论断找证据"的检索型工具。

### 目标

1. **读者视角**：用户从一组固定人格读者里选「今天请谁来读」，该读者**读完整篇**，回一封**第一人称、有体温、不打分、不替换原文**的信。换一个人读，在意的东西就变——评价标准 = 选谁来读。
2. **找引文**：用户选中一个论断/观点，联网检索 2–3 条**可信、带出处**的证据（名人名句、科学依据、历史事实等），点选复制、自行织入。

---

## 2. 功能范围

### 包含
- 后端两个新端点：`POST /assist/reader`、`POST /assist/cite`。
- 前端：「找引文」作为第 5 个划词动作进 `AssistPanel`；「读者视角」独立成新组件 `ReaderPanel.jsx`，右栏紧凑入口 + 浮层阅读区。
- 读者班底 5 人，写死为前端配置常量。
- 面板分组：改写类（缩减/同义/比喻/扩展/找引文）+ 反馈类（读者视角入口）。

### 不包含
- 自定义/新增读者人格（先锁定 5 人，组件配置预留可扩展，但不做编辑 UI）。
- 「收进信匣」的持久化存储（按钮留位，先不接后端）。
- 找引文结果的"怎么用进你这段"使用建议（每条只给原文 + 出处）。
- 读者视角注入 SOUL（刻意不注入，见 §5）。

---

## 3. 集成方式（方案 B：按交互形状分家）

两个新功能打破了现有"选中一段 → 替换"的模子，差异不同，故分别落位：

- **找引文** 仍是"选中论断 → 出候选 → 复制"，与「比喻」几乎同一模子（options + copy + 绕过 SOUL），天然属于 `AssistPanel`，复用现成划词管线，改动极小。
- **读者视角** 读整篇、不依赖选区、输出只读的长信、需先选人格——是全新交互形状，独立成 `ReaderPanel.jsx`，职责单一、可独立测试。

放弃方案 A（全塞进 AssistPanel，会撑爆其状态机）与方案 C（大重构为数据驱动面板，超出当前目标，YAGNI）。

## 4. 读者视角

### 4.1 读者班底（固定 5 人）

前端配置常量，每人 `{ key, glyph, name, en, care, lens }`：

| key | 字徽 | 名 | 透镜（在意什么 / care） | 谱系 |
|-----|----|----|------|------|
| poet | 诗 | 诗人 | 意象 · 节奏 · 语言质地 | 审美 |
| novelist | 叙 | 小说家 | 人物 · 场景 · 是"演"还是在"讲" | 叙事 |
| philosopher | 哲 | 哲学家 | 你这篇底下，真正在问什么 | 思想 |
| editor | 编 | 编辑 | 骨架 · 开头中段结尾立不立 | 结构 |
| debater | 辩 | 辩论家 | 立论 · 逻辑的漏洞 | 思辨 |

哲学家是**深化**（追问、把具体上升为普遍），辩论家是**反驳**（抓漏洞、要求论断站得住），两者区分明确。

### 4.2 前端

- 新组件 `ReaderPanel.jsx`，置于 `write-right` 列、`AssistPanel` 与 `DraftPanel` 之间（或其下），独立可折叠，与现有面板风格一致（`var(--card-grad)` / 字徽印章 / Fraunces+Noto Serif SC）。
- 右栏内只放**紧凑入口**：标题「今天，请谁来读？」+ 5 个字徽 + 一行 care。
- 点选读者 → 「<读者> 正在读…」过渡态（跳动墨点）→ 信件以**浮层专注阅读区**呈现（参照 `MoodCard.floating` 居中浮层样式）。信件较长（约 400–800 字），不适合塞进 300px 窄栏，故用浮层。
- 信件视图：印章落下动画 + 称呼 + 正文逐段浮现 + 拟人化落款（如「— 一个陪你多想一层的人 / 读罢·仍在想」）。关键判断句可带轻量下划线高亮。
- 操作：「再读一遍」（重新请求同一读者）、「换个人读」（回选择）、「收进信匣」（留位，先 no-op 或隐藏）。
- 信**只读，不替换、不写回**编辑器原文，不进撤回栈。
- 视觉与交互参照已确认的设计稿：`design-samples/读者视角-阅读区.html`。

### 4.3 后端

新端点 `POST /assist/reader`：

- 入参：`{ title: str, content: str, persona: str }`（persona ∈ 5 个 key；非法值回 400）。
- **不注入 SOUL**：读者要像一个不认识作者的真实读者，凭文章本身反应才诚实。
- 模型：`claude-opus-4-8`（共情与思辨都吃推理）。
- `max_tokens ≈ 1400`。
- 输出：`{ "letter": <单段信文> }`。
- system prompt 按 persona 取一段人格设定，模板：
  > 你是<人格名>，<一句人格刻画>。读一篇文章时，你只在意<透镜>。现在请你读完下面这篇，然后像一个真实的人，给作者本人写一封第一人称的信：有体温，不打分，不逐句批改，抓住真正打动你、或硌着你的地方来说；可以点名某个具体句子。结尾不必强行总结。约 400–800 字。只输出信的正文，不要加标题或前缀。
- user：`标题：<title>\n\n正文：\n<content>`。
- 错误：复用现有 502 降级（`AI 调用失败，请稍后再试`）。
- 可不复用 `_assist_call`（它面向 single/options 文本变换），单独写一个轻量 handler 即可。

## 5. 找引文

### 5.1 前端

- 在 `AssistPanel` 的 `ACTIONS` 增加第 5 项 `{ key:'cite', label:'📚 找引文', kind:'options', copy:true }`，走与「比喻」相同的 options + 复制路径。
- 选中论断（≥4 字）→ loading 文案「正在查证据…」（联网会慢几秒）→ 出 2–3 条证据卡。
- 每条卡 = **证据原文 + 出处**（出处含来源名与链接）。点选复制全文，自行粘回织入。
- 空结果文案：「没查到能确证出处的证据，换个说法或缩小论断再试。」

### 5.2 后端

新端点 `POST /assist/cite`：

- 入参：`{ text: str, context: str }`。
- **挂 Anthropic web_search 服务端工具**，只采纳搜索结果能支撑、查得到出处的证据。
- 模型：`claude-sonnet-4-6`。
- **不注入 SOUL**（证据是客观的）。
- prompt 硬约束：为这个论断找 2–3 条不同类型的证据（名人名句 / 科学依据 / 历史事实皆可），每条必须有真实可查的出处；**宁缺毋滥，绝不编造**名言、数据或年份；无法确证就不给。
- 输出：解析模型带 citations 的回答 → `{ "options": [ { "quote", "source", "url" } ] }`（最多 3 条）。
- 错误：502 降级；检索为空时返回空 options（前端显示空结果文案）。

### 5.3 实现风险（须在实现计划中处理）

- 现有 `_assist_call` 是单轮 `messages.create`；web_search 需要**工具使用循环 + 引用块解析**，因此 `cite` 必须**独立 handler**，不复用 `_assist_call`。
- web_search 工具的确切类型名、调用方式、`citations` 返回结构，**在写实现计划/实现时按 claude-api 参考核实**（勿凭记忆硬编码）。
- 联网检索有延迟与失败率，前端 loading/错误态要做足。

## 6. 文件改动

### 后端
- `backend/main.py`：新增 `AssistReaderRequest`、`POST /assist/reader`（含 5 段 persona system prompt 常量）、`AssistCiteRequest`、`POST /assist/cite`（含 web_search 工具循环 + 引用解析）。

### 前端
- `frontend/src/api.js`：新增 `assistReader({title,content,persona})`、`assistCite({text,context})`。
- `frontend/src/components/AssistPanel.jsx`：`ACTIONS` 加入「找引文」（options + copy）。
- `frontend/src/components/ReaderPanel.jsx`：**新建**，读者入口 + 浮层信件阅读区。
- `frontend/src/pages/Write.jsx`：在 `write-right` 挂载 `ReaderPanel`（传 `title` / `plainText`）。
- `frontend/src/App.css`：`ReaderPanel` 与信件浮层样式（沿用既有 token 与印章/信纸质感）。

## 7. 测试

- 后端：`POST /assist/reader` 各 persona 返回非空 letter；非法 persona 回 400；空 content 回 400。`POST /assist/cite` 返回结构为 `{options:[{quote,source,url}]}`；检索失败/为空时优雅降级（mock 掉 Anthropic 调用断言形状与降级路径，不打真实网络）。
- 前端：以现有手动/交互验证为主（仓库前端无自动化测试基建）。

## 8. 取舍与已确认决策

- 读者立场：以共情解读为主（意义层 + 体验层 + 一点结构层），不走默认抬杠；抬杠由"选辩论家"显式触发。
- 人格机制：用户**选谁来读**（非随机、非自动匹配、非多人同出）。
- 反应形态：**一封第一人称的信**（非评分、非反应笔记）。
- 找引文可信度：**联网检索 + 标出来源**（非纯模型生成）。
- 引文卡内容：**原文 + 出处**（不含使用建议）。
- 信件长度：约 400–800 字（较初版 mockup 加长 2–3 倍）。
- 班底：固定 5 人，名字即「编辑」「辩论家」，不加修饰前缀。
