# PRD · 写作页迭代（自动保存 · 每日一问 · 今日心绪卡）

> 框架：prd-development skill · 状态：工程可交付 · 分支：`feature/writing-page-iteration`
> 上游依据：`writing-page-{problem-statement,jtbd,ost,prioritization}.md` + `writing-page-xhs-research.html`
> 更新：2026-06-08

---

## 1. Executive Summary

我们在「文字时光机」的写作页，为**以随笔为主、偶尔轻创作的个人写作者**，解决「写作页是个会丢内容、写完即走的哑输入框，落笔当下感受不到应用的分析价值」这一问题。本轮交付**自动保存（防丢红线）**、**每日一问（化解空白页）**，以及差异化亮点 **今日心绪卡**——保存后原地浮出一张「情感基调 + 关键词 + 一句 AI 回应」的卡片，并**持久化进该篇文章、详情页常驻可见**。预期让"落笔第一次有回声"，提升周写作天数与次日回访。

---

## 2. Problem Statement

**谁**：习惯用写字梳理自己、偶尔想打磨成作品的个人写作者（随笔为主、不为发表、在意隐私不喂 AI、Mac 桌面）。

**什么问题**：写作页是全应用最朴素的一块（`Write.jsx` 纯 textarea），且：
1. 无自动保存——刷新/误触即丢；
2. 打开是空白框、无引导——空白页焦虑；
3. 保存后直接 `navigate('overview')` 跳走——落笔当下感受不到画像/情感/热力分析，**所有智能都堆在写完之后**。

**为何痛**：JTBD 里最致命且**无人解决**的痛——"写完石沉大海的虚无"；这也是写作者用「备忘录/flomo」而非本应用的根因。

**证据**：
- 代码现状：`Write.jsx` 63 行纯 textarea，保存即跳走；其余页面（总览/画像/仓库/详情）都很丰富。
- 竞品研究（`writing-page-xhs-research.html`）：所有小红书写作类 App 写作页都是哑输入框，**没人做"写完即分析"闭环**；多款 App 把"不喂 AI"当卖点 → 隐私敏感。

---

## 3. Target Users & Personas

**主 persona · 梳理者**：写日记/随笔/碎想，不为发表，为理解自己；试过备忘录/flomo/Notion 但半途而废；在意"我的文字不被拿去训练 AI"。
**次 persona · 轻创作者**（同一人的另一面）：偶尔把随笔扩成短文/公众号草稿，想要体面的成果与分享。

**核心 JTBD（本轮聚焦随笔核心）**：低摩擦快速落字 / 随时停随时接不丢 / 落笔即回馈意义 / 回头看见自己的变化。

---

## 4. Strategic Context

- **产品目标**：从"能写"升级到"写完有回声、会想再来"——把应用独有的分析超能力前置到写作当下，建立 vs. 备忘录/flomo 的差异化护城河。
- **竞争格局**：竞品写作页普遍有 自动保存 / 专注 / 导出长图 / 码字数据，但**无人做写作-分析闭环**；"不喂 AI"是写作者高频诉求。
- **Why now**：写作页是日常入口、却是当前最弱一环；其余分析能力（情感/画像/时间线）已就绪，闭环的"原料"现成，正是接起来的时机。
- **约束**：Phase 0 不大重构（FastAPI 同源单进程 + React）；用户自带 Key、本地优先；设计稿优先。

## 5. Solution Overview

### Phase 1 · 地基 + 空白页（先发）
**① 无感自动保存**
- 写作时内容实时存为草稿：前端 IndexedDB（即时、防刷新丢）+ 节流同步到后端草稿（接「半成品仓库」）。
- 重进写作页自动恢复未提交内容；正式「保存」后清理该草稿。

**② 每日一问（最轻形态）**
- 打开写作页时，标题区上方显示一句**动态欢迎语 + 一个轻选题**（按时段 + 写作画像生成）。
- 纯引导、不强制；点选题可把它带入正文起手；可忽略直接写。

### Phase 2 · 亮点（dogfood 验证）
**③ 今日心绪卡**
- 点「保存」后**不跳走**，正文下方/底部**原地浮出**一张卡：
  - **情感基调**（emoji + 一句话）← 本地 `compute_emotion_breakdown` 秒出
  - **关键词**（3-5 个）← 本地关键词提取秒出
  - **一句 AI 回应** ← 走用户自带 Key **异步**生成，先占位"正在读你的文字…"，回来再填
- 卡片可 **[知道了]**（收起）/ **[存为卡片]**（接 ljg-card 调性导出图）。
- **持久化**：卡片写入该篇 essay 的 `mood_card` 字段；**详情页 `EssayDetail` 常驻渲染**。
- **更新策略：最新一张覆盖**——每次保存重新生成、只留最新，始终反映当前正文。
- 隐私显性化：卡片 AI 回应区标注「这句只走你自己的 Key · 你的文字不被用于训练」。

## 6. Success Metrics

| 类型 | 指标 | 现状 → 目标 |
|---|---|---|
| **主指标** | 周写作天数（weekly active writing days） | 建基线 → ↑ |
| **主指标** | 保存后留存动作率（点开/保留心绪卡，未立即离开） | — → **>50%** |
| 次指标 | 次日回访写作率 | 建基线 → ↑ |
| 次指标 | 草稿恢复成功率（重进写作页内容不丢） | — → ~100% |
| 护栏 | 保存到"情感基调+关键词"出现的延迟 | — → <300ms（本地） |
| 护栏 | 自动保存不打断输入（无卡顿/无光标跳动） | 不回归 |

> 验证方式：单人 dogfood 1-2 周（先设计稿→认可→MVP→自用），看主观"写完有回声、想再写" + 上述行为指标。

## 7. User Stories & Requirements

### Epic Hypothesis
> 我们相信，为随笔写作者在写作页加上**自动保存 + 保存即回馈的今日心绪卡**，会提升周写作天数与次日回访，因为当前写作页会丢内容、且落笔当下毫无意义回声。以"保存后留存动作率 >50%"与 dogfood 主观感受验证。

### Phase 1

**S1 · 自动保存草稿**
> 作为写作者，我希望写的内容自动保存，这样刷新或误触都不会丢。
- [ ] 正文/标题变更后节流（~1.5s）写入 IndexedDB
- [ ] 节流（~5s 或失焦）同步到后端草稿（复用 vault 草稿通道）
- [ ] 自动保存全程**不打断输入**：无光标跳动、无 re-render 闪烁
- [ ] 顶部显示轻量状态：「已自动保存 · HH:MM」/「保存中…」

**S2 · 草稿恢复**
> 作为写作者，我重新打开写作页时希望接着上次写。
- [ ] 进入写作页若存在未提交草稿，自动恢复标题+正文
- [ ] 正式「保存」成功后，清除该草稿，避免重复恢复
- [ ] 从仓库「继续写」进入时，prefill 与草稿恢复不冲突（草稿优先级规则明确）

**S3 · 每日一问**
> 作为面对空白页的写作者，我希望打开就有一句引导。
- [ ] 写作页顶部显示动态欢迎语（按时段：清晨/午后/深夜不同）
- [ ] 附一个轻选题（结合写作画像；无画像时回退到通用选题池）
- [ ] 点选题 → 作为起手提示进入正文；可一键忽略
- [ ] 不强制、不拦截，不影响直接开写

### Phase 2

**S4 · 保存即浮出心绪卡**
> 作为写完一段的写作者，我希望立刻看到这段文字透露的状态。
- [ ] 点保存后**不跳页**，原地浮出心绪卡
- [ ] 情感基调 + 关键词 **<300ms 本地出**（`compute_emotion_breakdown` + 关键词提取）
- [ ] AI 回应区先占位「正在读你的文字…」，异步返回后填入
- [ ] AI 回应失败/无 Key：优雅降级，仅显示本地基调+关键词 + 一句兜底文案
- [ ] 卡片含 [知道了]（收起）与 [存为卡片]（导出图）

**S5 · 心绪卡持久化 + 详情页常驻**
> 作为写作者，我以后点开这篇文章，还能看到当时的心绪卡。
- [ ] 心绪卡随 essay 持久化（`mood_card` 字段）
- [ ] `EssayDetail` 文章详情页常驻渲染心绪卡
- [ ] **最新一张覆盖**：重新保存 → 重算覆盖；详情页永远显示最新
- [ ] 心绪卡数据结构化，供情感时间线/画像复用（不必重算全文）

### Edge cases
- 极短正文（<10 字）：跳过 AI 回应，仅本地基调或提示"再多写一点"。
- 无网络/无 Key：本地卡正常出，AI 区降级。
- 连续多次保存：AI 调用去抖，只为最终态生成，避免浪费 token。
- 旧文章无 `mood_card`：详情页不显示卡（或提供「生成心绪卡」按钮，后置）。

## 8. Out of Scope（明确不做）

- 强制小黑屋 / 重排版引擎 / 网文工作区（大纲·角色·伏笔·敏感词·取名）——不契合随笔
- 心绪卡**历史多张**快照（本轮"最新覆盖"；留待"时光机"主题深化）
- 边写边实时分析（S1.3）、与过去呼应（S1.2）、版本快照（S2.2）——已排期后置
- 移动端适配（Mac 桌面优先）

## 9. Dependencies & Risks

**数据模型（后端 `backend/main.py` · Essay）**
- 新增列 `mood_card TEXT`（JSON），沿用现有「缺列即 `ALTER TABLE ADD COLUMN`」幂等迁移模式。
- JSON 结构：
  ```json
  {
    "tone": "疲惫里有一点松动", "tone_emoji": "🌙",
    "keywords": ["焦虑","截止","妈妈"],
    "ai_reply": "你写了三次'终于'……", "ai_reply_status": "ok|pending|skipped|error",
    "generated_at": "2026-06-08T23:40:00", "model": "claude-..."
  }
  ```

**接口（`frontend/src/api.js` / 后端路由）**
- `POST /essays`、`PUT /essays/{id}`：保存时同步算本地心绪（tone+keywords）写入 `mood_card` 并随响应返回。
- `POST /essays/{id}/mood-reply`：用用户 Key 异步生成那句回应，回填 `mood_card.ai_reply`。
- 草稿：复用/扩展 vault 草稿通道（前端 IndexedDB 为主，后端为备份）。

**依赖**
- 本地关键词提取：确认词云是否已有可复用的关键词逻辑；若无，引入轻量分词（jieba）——评估是否违反 Phase 0「不大重构」（应属轻量依赖，可接受）。
- AI 回应：依赖用户已配置自带 Key 的链路。

**风险与缓解**
- *自动保存引起输入卡顿/光标跳* → 节流 + 非受控/局部更新，提交前充分 dogfood。
- *心绪卡显得冒犯/不准* → 文案走"陪伴"而非"评判"；可关；dogfood 调教措辞。
- *AI 延迟/失败拖累体验* → 本地先出、AI 异步补、失败优雅降级。
- *token 成本* → 去抖、仅最终态生成、极短文跳过。

## 10. Open Questions

1. 关键词提取：复用词云现有逻辑，还是新引入 jieba？（待看代码确认）
2. AI 回应的措辞风格与长度上限？（建议 ≤40 字、第二人称、不评判）——dogfood 中定。
3. 「存为卡片」直接复用 ljg-card 哪一种 mold？（建议 -b 大字附件卡或 -i 信息图）
4. 草稿与「半成品仓库」的边界：自动草稿何时"毕业"成仓库半成品？（建议：仅手动保存过的进入仓库）
